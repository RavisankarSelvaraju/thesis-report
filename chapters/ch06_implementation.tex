\chapter{Implementation} \label{chap:implementation}

To test the proposed methodology, the approach discussed in Chapter~\ref{chap:methodology} is implemented in the Coyote~3 rover software system and tested in a lab environment. This chapter provides the details of the rover-specific software implementation, the contact velocity odometry library and the modifications in the \gls{inekf} library. 

\section{Software Framework}

The proposed approach is implemented as a library which depends on the \gls{inekf} implementation developed by Ross Hartley~\cite{REPO-inekf-github}. The developed library is called \gls{cvo}. The Coyote~3 software system uses the ROCK~\cite{SW-rockrobotics} framework for development and deployment of its libraries. The functionalities can be developed as libraries independent of the ROCK framework and can later be integrated into the oroGen component based framework. The component based systems are built by Orocos Real Time Toolkit (Orocos RTT)~\cite{SW-orocos-rtt}. The Orocos RTT have a state machine representation using Hook methods as mentioned in rock-robotics documentation~\cite{SW-rockrobotics}. These methods are explained in Section~\ref{subsec:hookMethods}. Since this thesis implementation does not modify or add functionalities to the ROCK framework, the details on these systems are not provided. Readers who are interested in these systems can refer the rock-robotics documentation~\cite{SW-rockrobotics} for further details. The oroGen components that are relevant to this thesis implementation are shown in Figure~\ref{fig:component_map_implementation}. The data path between the components is also shown in the figure.

\section{Contact Velocity Odometry Estimator Component}

The purpose of the \gls{cvoe} component is to connect the implemented \gls{cvo} library to the ROCK component framework and obtain the necessary data for the \gls{cvo} library. The oroGen component framework uses a \texttt{.orogen} file which contains component description which includes the input ports, output ports, properties, and transformations necessary for the component. This file is provided in Listing~\ref{code:orogenfileCVOE}. This component is connected with other components via input and output ports. The input ports relevant to this implementation are: \texttt{measurement\_imu}, \texttt{joint\_state}, \texttt{pose\_imu}. The output ports relevant to this implementation are: \path{pose_body_estimated} and \path{pose_body_estimated_local}. There are three main data types that are used, the \texttt{IMUSensors}, the \texttt{RigidBodyState} and the \texttt{Joints}. The \texttt{measurement\_imu} port uses \texttt{IMUSensors} data type, the \texttt{joint\_state} uses \texttt{Joints} data type and \texttt{pose\_imu}, \path{pose_body_estimated} and \path{pose_body_estimated_local} use \texttt{RigidBodyState} data type. Additional debug ports are added to inspect the selected contact points, estimated biases, calculated innovation, and state covariance matrices. 

The properties defined in the \texttt{.orogen} file are configured using a YAML configuration file provided in Listing~\ref{code:YAMLConfigfileCVOE}. This helps to change the \gls{inekf} configuration without rebuilding the \texttt{C++} project. Some robot-specific properties such as path to the Coyote~3 \gls{urdf} model, the coordinate frame names for world, body, wheel, and contact frames, the bias estimation and debug flags, the noise parameters and the threshold values for slip rejection and multi-contact estimation are included in the configuration file.

Apart from the data flow and the configuration, the \gls{cvoe} component also performs several preprocessing steps before calling the \gls{cvo} library. 
These include \gls{imu} and joint encoder data preprocessing, IMU frame transformation, initial gyroscope bias estimation, and kinematic calculations. These implementation details are explained in the following sections.


    \subsection{OroGen Component Hook Methods} \label{subsec:hookMethods}

    The oroGen component framework uses a state machine representation for the components. Every component can define the following hook methods: \texttt{configureHook}, \texttt{startHook}, \texttt{updateHook}, \texttt{errorHook}, \texttt{stopHook}, \texttt{cleanupHook}. Each method is called when the component transitions from one state to another. Components can be in one of the following states: \texttt{PreOperational}, \texttt{Stopped}, \texttt{Running}, \texttt{Exception}. Detailed information is available in the rock-robotics documentation~\cite{SW-rockrobotics}. A brief explanation of the methods necessary to understand the thesis implementation process flow is explained here. 

    The \texttt{configureHook} method is the first hook method called after the construction of the component. This method reads the properties that are defined in the \texttt{.orogen} file and set the class member variables of the component. Sanity checks on the property variables are also done in this method. Next, the \texttt{startHook} is called, for this implementation, \texttt{startHook} is not used. The \texttt{updateHook} is called after the \texttt{startHook}. This \texttt{updateHook} is continuously called until the state changes from \texttt{Running} to \texttt{Stopped}. The \texttt{errorHook}, \texttt{stopHook} and \texttt{cleanupHook} are also not used in this implementation. The preprocessing steps and triggering the \gls{cvo} library were done in the \texttt{updateHook} which was triggered at 100Hz. This implementation technique does not align the input data streams, so the implementation is moved into the \gls{imu} transformer callback method. This is further explained in Section~\ref{subsec:StreamAlignerandTransformerCallbacks}. 

    \subsection{Data Stream Alignment and Transformer Callbacks} \label{subsec:StreamAlignerandTransformerCallbacks}

    The \gls{imu} and joint encoder data arrive at the \gls{cvoe} component from different components, which can have different computation times. The data samples received at the \gls{cvoe} component may not arrive at the same order in which they were sensed. Hence, ROCK provides a \texttt{C++} library called \textit{stream aligner} which process the data streams in the correct temporal order. A data sample is sent to the corresponding callback methods in the component only when the \textit{stream aligner} decides that no samples from the other streams that are aligned can arrive with an earlier timestamp than the current sample. 

    This \textit{stream aligner} can be defined in the \texttt{.orogen} file as \verb|align_port <port name>, <period>| as shown in Listing~\ref{code:orogenfileCVOE}. In the \gls{cvoe} component, it is defined inside the transformer block for the \texttt{measurement\_imu} port and \texttt{joint\_state} port. More details on the \textit{stream aligner} can be found in rock-robotics documentation~\cite{SW-rockrobotics}. This creates two callback methods: \texttt{measurement\_imuTransformerCallback} and \texttt{joint\_stateTransformerCallback}. These callbacks are used for the data preprocessing steps and triggering the \gls{cvo} library. The initial gyroscope bias estimation, and filter propagation step are performed in the \texttt{measurement\_imuTransformerCallback} method. Wheel and contact kinematics are calculated inside the \texttt{joint\_stateTransformerCallback} method.     

    \subsection{Initial Gyro Bias Estimation}

    When the \gls{imu} data is received in the \texttt{measurement\_imuTransformerCallback} method, the first step is to calculate the bias in the gyroscope. The \texttt{IMUSensors} type contains the linear acceleration from the accelerometer and the angular velocity from the gyroscope. The rover is stationary when this step is performed. When the rover is stationary, the angular velocity that the gyroscope senses is the sensor bias. The bias estimation is done for a fixed duration which is defined in the \texttt{.orogen} file. The average of the gyroscope measurements in this duration is estimated as the gyroscope bias. Gyroscope measurements with norm below a threshold are considered for the estimation to avoid measurements recorded when the rover is moving. This bias is subtracted when the measured angular velocity is used in the \gls{inekf}. Initially, the estimated gyroscope bias is expressed in the \gls{imu} reference frame. The pose estimation calculations use the body frame of reference, so the bias estimate is transformed to the body frame using the static transformation from the \texttt{pose\_imu} input port. 

    \subsection{Wheel and Contact Kinematics} \label{subsec:wheelAndContactKinematics}

    The joint state data is received as \texttt{joint\_state} datatype in the \texttt{joint\_stateTransformerCallback} method. The wheel speeds are calculated from this joint state data. It is stored in a class member variable to be used when triggering the \gls{cvo} library. The joint state data is also used to calculate the transformation between different frames that are needed in the \gls{cvo} library. The transformer broadcaster component shown in Figure~\ref{fig:component_map_implementation} provides the transformation between the frames mentioned in the \texttt{transformer do} block of the \texttt{.orogen} file. The transformation names mentioned in the \texttt{.orogen} file are assigned in the configuration YAML file provided in Listing~\ref{code:YAMLConfigfileCVOE}. The \gls{cvoe} components get the transformation as \texttt{Eigen::Affine3d} datatype using the \texttt{transformer}. These transformations are used to express the poses of spoke tips, and wheel frames with respect to the body frame and the poses of contact points with respect to the fixed wheel frames. The fixed wheel frames are included in the robot \gls{urdf} model to define a fixed frame of reference to get the position vectors of spoke tips relative to a non-rotating frame of reference. These frames are added at the location of motors. This fixed wheel frames have the same pose as the initial pose of the rotating wheel frame.

    \subsection{Contact Velocity Odometry Library Interface}

    The \gls{cvo} library does a two-step process, propagation and correction. Both processes are triggered from the \gls{cvoe} components inside the \texttt{measurement\_imuTransformerCallback}. Once the initial gyroscope bias is estimated, the \texttt{propagate} method of the \gls{cvo} is called with the \gls{imu} data expressed in the body frame. After the propagation step, the correction step is done by calling the \texttt{correctContactVelocity} method with the data calculated in the previous sections. Before triggering the \gls{cvo} library methods, an object of the \texttt{Odometry} class is created and an initial state and noise parameters are assigned using \texttt{initialize} method. The \texttt{setRobotConfiguration} method is used to set robot-specific parameters such as number of wheels, number of spokes per wheel and the robot \gls{urdf} model. 

\section{Contact Velocity Odometry Library}

    The \gls{cvo} library is developed to work independent of the \gls{cvoe} component and the ROCK framework. The proposed approach discussed in the Methodology Chapter~\ref{chap:methodology} is implemented in this library. This library is developed as a class called \texttt{Odometry} and the \gls{cvoe} component uses an object of this class to utilize this library's functionalities. This library provides getters for filter state, state covariance, Kalman gain, and innovation vector and setters for initial filter state, noise parameters. These methods are provided to interact with the \gls{inekf} library. 
    
    The main methods used during the runtime of the library are \path{propagate} and the \path{correctContactVelocity}. The \path{propagate} method, gets the \gls{imu} data from the \gls{cvoe} component and use that data to trigger the \gls{inekf} library's \path{Propagate} method. This method and its functionalities unmodified from the original \gls{inekf} implementation. The \path{correctContactVelocity} method uses \path{computeVectorKinematics}, \path{computeExpectedContactVelocity}, \path{computeMeasuredContactVelocity} methods to calculate necessary data to trigger the \gls{inekf} library's \path{CorrectContactVelocity} method. The following section details these helper methods and other implementation details in the \gls{cvo} library.
        
    \subsection{Kinematic Data Construction}
    
    The \gls{inekf} library provides the \texttt{Kinematics} data structure, and \texttt{vectorKinematics} is used to store a vector of these structures. The data structure is shown in Listing~\ref{code:vectorKinematicsDatastructure}. This data structure is used to store the transformations calculated in Section~\ref{subsec:wheelAndContactKinematics}. The \texttt{Eigen::Affine3d} transformations received from the \gls{cvoe} component are converted to \texttt{Eigen::Matrix4d} and stored in \texttt{vectorKinematics} objects. 
    
    Three kinematic vectors are constructed: \path{potential_contact_to_body_kinematics}, \path{potential_contact_to_wheel_kinematics}, and \path{wheel_to_body_kinematics}. These vectors are constructed inside the \path{computeVectorKinematics} method. \path{wheel_to_body_kinematics} stores the transformations from the fixed wheel frames to the body frame. \path{potential_contact_to_body_kinematics} stores the transformations from the contact points, also referred to as \glspl{pc}, to the body frame. \path{potential_contact_to_wheel_kinematics} stores the transformations from the \glspl{pc} to the fixed wheel frames. The \texttt{id} field helps to associate each transformation with the corresponding wheel or contact point. For the contact-related kinematic vectors, the \texttt{id} field is calculated using the global contact index with the formula presented in Equation~\ref{eq:globalContactIdx}.
    
    \begin{equation}
        i_{\mathrm{global}} = i_{\mathrm{wheel}} \cdot N_{\mathrm{contacts}} + i_{\mathrm{contact}}
        \label{eq:globalContactIdx}
    \end{equation}
        
    where $i_{\mathrm{global}}$ is the global contact index, $i_{\mathrm{wheel}}$ is the wheel index, $N_{\mathrm{contacts}}$ is the number of contacts per wheel, and $i_{\mathrm{contact}}$ is the contact index within the wheel. The robot \gls{urdf} model is parsed to extract the rotation axis of each wheel joint. This axis is multiplied with the wheel speed from the joint data to construct the wheel angular velocity vector.
    

    \begin{lstlisting}[language=C++,
                   directivestyle={\color{black}}
                   emph={int,char,double,float,unsigned},
                   emphstyle={\color{blue}},
                   caption={Kinematics data structure}, label={code:vectorKinematicsDatastructure}]
    class Kinematics {
    public:
        EIGEN_MAKE_ALIGNED_OPERATOR_NEW
        Kinematics( int id_in, 
                    Eigen::Matrix4d pose_in, 
                    Eigen::Matrix<double,6,6> covariance_in) : 
                    id(id_in), 
                    pose(pose_in), 
                    covariance(covariance_in) { }

        int id;
        Eigen::Matrix4d pose;
        Eigen::Matrix<double,6,6> covariance;
    };        
    \end{lstlisting}
       
    \subsection{Potential Contact Velocity Calculation}
    
    The poses stored as \texttt{Kinematics} objects are used in the \gls{pcv} calculation. As mentioned in the Methodology Chapter~\ref{chap:methodology}, we have predicted and measured \gls{pcv} calculations. The equations and explanations behind these calculations are already discussed in the Methodology Chapter. This section provides the details on the implementation of those equations in the library. 
    
    The predicted \gls{pcv} is computed in the \path{computeExpectedContactVelocity} method. This method uses the kinematics from the \gls{pc} frame to the body frame (\path{potential_contact_to_body_kinematics}), the bias-corrected angular velocity of the body (\texttt{omega\_corrected}), the rotation from the world frame to the body frame (\path{R_world_to_body}), and the body velocity expressed in the world frame (\path{v_body_in_world}). The result is stored in \path{expected_contact_velocity_in_body}. Inside this method, the \gls{pcv} vector is calculated by constructing a body-frame twist and changing its reference point to each \gls{pc}. The \path{v_body_in_world} vector is rotated to obtain the body velocity expressed in the body frame, \path{v_body_in_body}.
    
    A \texttt{KDL::Twist} is constructed using this velocity vector and the \texttt{omega\_corrected} vector. This constructed twist is stored in the \texttt{twist\_body\_in\_body} variable and is expressed in the body frame.. The reference point of this twist is changed to the \gls{pc} using the \texttt{.RefPoint} method from the \gls{kdl} library. The resulting twist is stored as \texttt{twist\_at\_pc}. The twist at the \gls{pc} is used to extract the linear velocity vector, which is then stored in an \texttt{Eigen::VectorXd}. As discussed in the Methodology Chapter~\ref{subsec:predictedPCV}, the calculated velocity vectors are negated and each vector is stored in \path{expected_contact_velocity_in_body} using the corresponding \texttt{id} from \path{potential_contact_to_body_kinematics}. This is done for all \glspl{pc} that is stored in the (\path{potential_contact_to_body_kinematics}) vector. The background required to understand twist representation and reference point changes is explained in Background Chapter Section~\ref{subsec:transformingTwists}.
    
    The measured \gls{pcv} is computed in the \path{computeMeasuredContactVelocity} method. The kinematics from the \gls{pc} frame to the fixed wheel frame (\path{potential_contact_to_wheel_kinematics}) and the wheel angular velocity vector (\path{omega_wheel}) are used in this method. The \gls{pcv} is initially calculated in the fixed wheel frame of reference using Equation~\ref{eq:MeasuredPCVCalculation}. The resulting velocity vector is then transformed to the body frame using the rotation from the fixed wheel frame to the body frame extracted from the fixed wheel to body kinematics (\path{wheel_to_body_kinematics}). Similar to the predicted \gls{pcv}, the measured \gls{pcv} is stored in an \texttt{Eigen::VectorXd} using the corresponding \texttt{id} from \path{potential_contact_to_wheel_kinematics}. This is done for all \glspl{pc} that is stored in the (\path{potential_contact_to_wheel_kinematics}) vector. Both the predicted and measured \gls{pcv} vectors are available in the \path{correctContactVelocity} method after the \path{computeExpectedContactVelocity} and \path{computeMeasuredContactVelocity} methods are called. 
    
    \subsection{Innovation and Jacobian Calculation}  
    
    Once the \gls{pcv} vectors are calculated, a data structure is created to store the quantities associated with each spoke is created. This data structure is called \texttt{SpokeData} and is shown in Listing~\ref{code:spokeDataStructure}. A nested \texttt{for} loop is created to iterate through all the wheels and spokes. A \texttt{std::map} of \texttt{SpokeData} is created with the global contact index as the key. This map is updated inside the loop with the calculated innovation, Jacobian, and noise matrices. Inside the for loop, the \gls{pcv} vectors are extracted for the corresponding global contact index. The innovation is calculated by subtracting the predicted \gls{pcv} from the measured \gls{pcv} as shown in Equation~\ref{eq:pcvResidual}. 
    
    \begin{lstlisting}[language=C++,
                   directivestyle={\color{black}}
                   emph={int,char,double,float,unsigned},
                   emphstyle={\color{blue}},
                   caption={Spoke data structure}, label={code:spokeDataStructure}]
    struct SpokeData {
        
        // 1 if in contact, 0 if not in contact, -99 if not determined yet
        double in_contact = -99; 
        
        // 1 if non-slipping, 0 if slipping, -99 if not determined yet
        double non_slipping = -99; 
        
        Eigen::Vector3d innovation;
        Eigen::MatrixXd jacobian;
        Eigen::Matrix3d noise;
    };
    \end{lstlisting}
        
    The Jacobian is constructed based on the Equation~\ref{eq:MeasurementJacobian-3}.The rotation block is filled with the negated skew-symmetric matrix of the body velocity expressed in the body frame. The velocity block is the negated identity matrix. The gyroscope bias block is filled with the negated skew-symmetric matrix of the position vector from the body frame to the \gls{pc} expressed in the body frame. After the Jacobian is constructed, The measurement noise matrix is constructed by treating the measurement noise parameter as a standard deviation and squaring it to obtain the covariance. This measurement noise parameter is defined in the \texttt{.orogen} file. The \texttt{spokeData} map is updated with the calculated innovation, Jacobian, and noise matrices.     
    
    \subsection{Slip Rejection and Contact Selection}
    
    The Mahalanobis distance is calculated as shown in the Equations~\ref{eq:innovationCovariance} and \ref{eq:mahalanobisDistanceSquared}. For each spoke, the squared \gls{md} is calculated and compared with the chi-squared threshold defined in the \texttt{.orogen} file. The chi-squared threshold is used for the slip rejection step. In this implementation, the residual for each \gls{pc} is a three-dimensional vector. Therefore, the squared \gls{md} is compared with a chi-squared threshold with three degrees of freedom.
    
    If the squared \gls{md} is below this threshold, the measured \gls{pcv} is considered consistent with the predicted \gls{pcv}. Hence, the spoke is considered to be non-slipping. If the squared \gls{md} is above this threshold, the measured \gls{pcv} is considered inconsistent with the predicted \gls{pcv}, and the spoke is considered to be slipping. For a three-dimensional residual, the commonly used 95\% chi-squared threshold is 7.815.
    
    The \gls{pc} selection is done based on two methods: the \gls{lf} and the \gls{md}. The method used to update the \gls{inekf} is decided based on the configuration flag \texttt{estimate\_contacts}. If this flag is true, the \gls{md} method is used to select the \gls{pc}. If this flag is false, the \gls{lf} method is used to select the \gls{pc}. 
    
    The \gls{md} method of contact estimation selects the spoke with the lowest squared \gls{md} value and also checks the spoke with the second-lowest squared \gls{md} value. If the difference between the lowest and the second-lowest squared \gls{md} values is below the \texttt{contact\_md2\_threshold}, both spokes are selected as \glspl{pc}. Otherwise, only the spoke with the lowest squared \gls{md} is selected. In the experiments, this threshold value is configured as \texttt{0.002}.
    
    In the \gls{lf} method of contact estimation, initially, the spoke with the lowest \texttt{z} position is selected as the \gls{pc} then a threshold region is created in the \texttt{z} position of the spoke tip in the body frame. The threshold region is created by adding \texttt{0.02 m} to the \texttt{z} position of the selected spoke tip. Spokes with \texttt{z} position below this threshold are also selected as \glspl{pc}. The threshold value of \texttt{0.02 m} is experimentally selected in a flat and solid terrain. 
    
    Once the slip rejection and contact selection are done, the \texttt{SpokeData} map is updated with the contact and slip status of each spoke. 
    For updating the \gls{inekf}, the innovation, Jacobian, and noise matrices of the \glspl{pc} which are non-slipping and in contact are extracted from the \texttt{SpokeData} map and stored in \texttt{Eigen::VectorXd}, \texttt{Eigen::MatrixXd}, and \texttt{Eigen::MatrixXd}.
    
    When all the spoke data is processed, the \texttt{CorrectContactVelocity} method of the \gls{inekf} library is called with the extracted innovation, Jacobian, and noise matrices. The \gls{inekf} library is extended with this new method to handle the contact velocity correction step. The \texttt{correctContactVelocity} method uses the \texttt{CorrectLeftInvariant} method to update the filter state and covariance. The updated filter state and covariance are stored in the \gls{cvo} library and can be accessed using the getter methods. If there are no spokes that satisfy the contact and slip conditions, the \texttt{CorrectContactVelocity} method is not called, and the filter state and covariance are not updated.
